# 报告生成 Prompt — {{PROJECT_NAME}}

## 使用说明

将所有走查轮次的 Markdown 结果合并后，粘贴到以下 Prompt 中。
在新的 Claude/Agent 会话中执行此 Prompt 即可生成最终报告。

**操作步骤：**
1. 将 `{{PROJECT_NAME}}`、`{{N}}`、`{{JOURNEY_STAGES}}`、`{{PERSONA_LIST}}`、`{{PERSONA_WEIGHTS}}`、`{{ANALYSIS_DIMENSIONS}}`、`{{FILTER_DIMENSIONS}}` 替换为实际值
2. 将 `[在此粘贴所有走查轮次的合并 Markdown]` 替换为真实走查数据
3. 将整个 Prompt 块复制到新的 Claude 会话中执行

````

包含 {{N}} 个场景的走查发现（多画像 × 多轮次）。

请基于这些走查数据，生成一份完整的、自包含的 HTML 格式体验分析报告。

### 报告要求

生成一个单文件 HTML 报告，包含以下四个核心板块。所有代码写在一个 `.html` 文件中，
不依赖任何外部 CDN、字体或 JavaScript 库——纯 HTML + 内联 CSS + 原生 JS。

#### Block 1: 体验温度地图 (Experience Temperature Map)

**可视化类型:** 热力图（Heatmap）

**X 轴:** 用户旅程阶段及子阶段
旅程阶段定义如下：
{{JOURNEY_STAGES}}

**Y 轴:** 画像 × 新老用户维度
画像列表：
{{PERSONA_LIST}}
每个画像拆分为「新用户」和「老用户」两行，Y 轴共 {{PERSONA_Y_COUNT}} 行。

**单元格颜色逻辑 (加权问题评分):**
- P0（严重问题）权重 = 4
- P1（有问题）权重 = 3
- P2（可接受偏差）权重 = 2
- P3（轻微瑕疵）权重 = 1
- 单元格得分 = Σ(每条问题 × 对应权重)
- 颜色映射:
  - 得分 0 → `#E8F5E9`（绿，健康）
  - 得分 1–3 → `#FFF9C4`（浅黄，关注）
  - 得分 4–7 → `#FFE0B2`（橙，有问题）
  - 得分 8–12 → `#FFCCBC`（橙红，严重）
  - 得分 13+ → `#FFCDD2`（红，危险）

**交互规范:**
- Hover 单元格: 弹出 tooltip，显示
  - 该格问题总数
  - 各严重度分布（P0: N 条，P1: N 条，P2: N 条，P3: N 条）
  - 若走查数据中有截图引用，显示缩略图（64×64px）列表，带步骤编号
- Click 单元格: 页面平滑滚动至 Block 4「全集问题数据库」，
  并自动激活对应画像 + 旅程阶段的筛选器，高亮相关行

**视觉要求:**
- 单元格最小尺寸 80×48px，文字显示问题总数（数字居中）
- 表头固定（sticky），支持横向滚动（当阶段数量多时）
- 图例在热力图右侧或下方，说明颜色含义

---

#### Block 2: 体验洼地分析 (Experience Valley Analysis)

**分析框架:** 从以下维度进行交叉变量分析
{{ANALYSIS_DIMENSIONS}}

**每个维度输出格式:**
- 维度标题（加粗大字）
- Top 3 关键洞察（Insight Card 形式）
  - 洞察标题（一句话结论）
  - 支撑证据（引用 1–2 条具体走查发现，注明画像 + 阶段）
  - 若走查数据包含截图引用，显示「用户实际看到的」截图（最多 2 张，带红色注释框标注问题位置）
  - 优化建议方向（简短，1–2 句）

**权重规则:**
画像样本权重如下（影响洞察的优先级排序）：
{{PERSONA_WEIGHTS}}
- 高权重画像的洞察优先展示
- 低权重画像的边缘 case 降级为「附注」，不作为主洞察
- 每个洞察 Card 右上角显示受影响的画像标签（带权重百分比）

**视觉要求:**
- Insight Card：白底，左侧 4px 色条（蓝色 = 发现，橙色 = 风险，绿色 = 机会）
- 截图展示区：灰色背景容器，图片 max-width: 480px，下方有 caption

---

#### Block 3: 链路断裂点分析 (Journey Breakpoint Analysis)

**分析目标:** 找出问题最集中的旅程阶段，识别「断裂点」

**主可视化: 旅程流程健康图**
- 横向流程图，每个阶段为一个节点方块
- 节点颜色:
  - 绿色（`#43A047`）: 该阶段加权得分 ≤ 3（健康）
  - 黄色（`#FDD835`）: 得分 4–8（关注）
  - 红色（`#E53935`）: 得分 > 8（断裂点）
- 节点内显示: 阶段名称 + 问题总数 + 最高严重度标记
- 节点间用箭头连接，箭头宽度可视化流量/重要性
- 断裂点节点有红色脉冲动画（CSS keyframes）

**行业维度叠加（如适用）:**
- 若走查数据涵盖多个行业类型，在流程图下方增加「行业对比视图」
- 对比参数型行业 vs 视觉型行业的断裂点差异
- 使用分组柱状图（纯 CSS + JS 渲染，无外部库）

**断裂点详情展开:**
- 点击红色/黄色节点，在节点下方展开详情面板
- 详情包含:
  - 该阶段 Top 3 问题（标题 + 严重度）
  - Before（现状）截图 vs After（理想态/竞品参考）对比区（若走查数据提供）
  - 涉及画像列表（哪些用户群在此阶段受影响最重）

**视觉要求:**
- 流程图容器横向滚动（overflow-x: auto）
- 健康/关注/断裂三态图例清晰标注
- 展开面板有平滑动画（CSS transition）

---

#### Block 4: 全集问题数据库 (Complete Issue Database)

**布局:** 左右分栏
- 左栏（60%）: 问题文字信息
- 右栏（40%）: 问题截图（带注释）

**左栏内容（每行展示）:**
- 问题 ID（格式：`[阶段缩写]-[序号]`，如 `F-01-003`）
- 问题标题（一句话描述）
- 严重度标签（P0/P1/P2/P3，颜色编码：P0=红，P1=橙，P2=黄，P3=灰）
- 受影响画像标签（多选，颜色标签）
- 旅程阶段标签
- 问题描述（2–4 句，说明发现了什么、用户体验到什么）
- 优化建议（1–3 条 bullet，简洁可执行）
- 「展开上下文」按钮 → 展开后显示该步骤前后相邻步骤的截图序列（横向 scroll）

**右栏内容（每行对应）:**
- 走查截图（若有）：max-height 200px，带红色方框标注问题区域
- 若无截图：灰色占位方块 + 「走查时未截图」提示

**筛选器（固定在顶部，sticky）:**
{{FILTER_DIMENSIONS}}
- 筛选器为多选 tag 形式，选中状态高亮（蓝色填充）
- 「清除筛选」按钮（右侧）
- 筛选结果实时更新（JS filter，无需重载）
- 显示当前筛选结果数量（「显示 12 / 47 条问题」）

**排序选项:**
- 默认: 按严重度降序（P0 → P3）
- 可切换: 按旅程阶段顺序 / 按受影响画像数量 / 按问题 ID

**行展开行为:**
- 默认折叠（仅显示标题行）
- 点击行展开完整内容
- 展开动画（max-height transition）

### 全局 HTML 技术规范

**文件结构:**
```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>{{PROJECT_NAME}} 体验走查报告</title>
  <style>/* 所有样式内联 */</style>
</head>
<body>
  <!-- 顶部导航 -->
  <!-- Block 1: 体验温度地图 -->
  <!-- Block 2: 体验洼地分析 -->
  <!-- Block 3: 链路断裂点分析 -->
  <!-- Block 4: 全集问题数据库 -->
  <script>/* 所有 JS 内联 */</script>
</body>
</html>
```

**顶部导航:**
- 固定顶部（position: fixed），白底 + 底部阴影
- 左侧: 项目名称 + 报告生成日期
- 右侧: 4 个锚点链接，点击平滑滚动（scroll-behavior: smooth）

**配色系统:**
```
主色调: #1565C0（深蓝，用于标题、链接、激活态）
辅助蓝: #42A5F5（中蓝，用于次级元素）
背景色: #F8FAFC（页面背景）
卡片色: #FFFFFF（内容区域）
边框色: #E1E8ED（分割线）
文字主色: #1A2332（正文）
文字次色: #5A6B7B（辅助说明）
严重度红: #E53935
严重度橙: #FB8C00
严重度黄: #FDD835
严重度灰: #9E9E9E
健康绿: #43A047
```

**字体:**
- 中文: system-ui, 'PingFang SC', 'Microsoft YaHei', sans-serif
- 数字/代码: 'SF Mono', 'Consolas', monospace

**响应式:**
- 断点 1200px: 两栏布局 → 单栏
- 断点 768px: 热力图横向滚动，隐藏次要列

**截图处理:**
- 若走查数据中截图以文件路径引用（如 `./screenshots/step_01.png`），使用相对路径引用
- 若截图为 base64 内联，直接嵌入 `<img src="data:image/png;base64,...">` 
- 若无截图，显示灰色占位块（`background: #ECEFF1; border: 2px dashed #B0BEC5`）

**可访问性:**
- 所有交互元素有 focus outline
- 颜色不作为唯一信息载体（配合图标或文字标签）
- 热力图单元格有 aria-label（描述该格内容）

**性能:**
- 问题数据库超过 50 条时，实现虚拟滚动（JS）或分页（每页 20 条）
- 图片懒加载（loading="lazy"）

---

### 输出格式

直接输出完整的 HTML 代码，从 `<!DOCTYPE html>` 开始，到 `</html>` 结束。
不要解释，不要 markdown 包装，直接输出可保存为 `.html` 的完整代码。
````

## 变量说明

| 变量 | 说明 | 类型 | 本次实例值 |
|------|------|------|------------|
| `{{PROJECT_NAME}}` | 项目名称，用于报告标题和文件名 | 字符串 | - |
| `{{N}}` | 走查场景总数（画像数 × 轮次数） | 整数 | - |
| `{{JOURNEY_STAGES}}` | 用户旅程阶段定义，含阶段名和子阶段列表 | Markdown 列表 | - |
| `{{PERSONA_LIST}}` | 画像列表，每个画像一行，含简短描述 | Markdown 列表 | - |
| `{{PERSONA_Y_COUNT}}` | Y 轴总行数（画像数 × 2，因为新/老用户各一行） | 整数 | - |
| `{{PERSONA_WEIGHTS}}` | 各画像的样本权重（百分比），用于分析时的重要性加权 | Markdown 表格 | - |
| `{{ANALYSIS_DIMENSIONS}}` | 体验洼地分析的交叉维度列表（如：画像维度、品类维度、行业维度等） | Markdown 列表 | - |
| `{{FILTER_DIMENSIONS}}` | 问题数据库的筛选维度（如：严重度、画像、阶段、品类、行业） | Markdown 列表 | - |

## 填写示例（Alibaba.com / Accio 走查实例）

以下为针对 Alibaba.com B2B 电商走查的变量填写示例，可直接替换上方变量使用：

```
{{PROJECT_NAME}} = Alibaba.com 全链路体验走查报告 2025-Q3

{{N}} = 15（5 画像 × 3 轮次）

{{JOURNEY_STAGES}} =
- 找 (Find): 首页导购 → 搜索 → 筛选 → 浏览结果
- 挑 (Select): 商品详情 → 供应商核实 → 比较
- 询 (Inquire): 联系供应商 → 询盘表单 → 后续跟进

{{PERSONA_LIST}} =
- small_retail: 小零售商，年采购额 < $50K，新手偏多，对 MOQ 敏感
- medium_brand: 中型品牌商，年采购额 $50K–500K，有采购经验
- large_brand: 大型品牌商，年采购额 > $500K，有专职采购团队
- service_provider: 服务商/代理，代客采购，熟悉平台操作
- manufacturer: 制造商自采，专业采购，主要关注原材料和工业品

{{PERSONA_Y_COUNT}} = 10

{{PERSONA_WEIGHTS}} =
| 画像 | 权重 | 说明 |
|------|------|------|
| small_retail | 40% | 平台最大用户群 |
| medium_brand | 25% | 核心付费用户 |
| large_brand | 15% | 高价值用户 |
| service_provider | 12% | 高频用户 |
| manufacturer | 8% | 小众但高专业度 |

{{ANALYSIS_DIMENSIONS}} =
- 画像维度：不同买家类型的体验差异
- 品类维度：标准品 vs 轻定制 vs 重定制的体验差异
- 行业维度：参数型行业（五金/电子）vs 视觉型行业（服装/家居）的体验差异
- 新老用户维度：首次访问 vs 注册用户 vs 付费会员的体验差异

{{FILTER_DIMENSIONS}} =
- 严重度: P0 / P1 / P2 / P3
- 旅程阶段: 找 / 挑 / 询
- 画像: small_retail / medium_brand / large_brand / service_provider / manufacturer
- 品类: 标准品 / 轻定制 / 重定制
- 行业: 参数型 / 视觉型
```

## 复用说明

### 固定部分（跨项目通用，不需修改）
1. 四大板块结构（温度地图、洼地分析、断裂点、问题数据库）
2. HTML 技术规范（文件结构、配色系统、字体、响应式、可访问性）
3. 各板块的交互规范（hover/click 行为、筛选逻辑、展开动画）
4. 严重度权重公式（P0=4, P1=3, P2=2, P3=1）

### 可变部分（每个项目实例化时替换）
1. `{{PROJECT_NAME}}` — 项目名称
2. `{{JOURNEY_STAGES}}` — 旅程阶段名称（不同产品的旅程不同）
3. `{{PERSONA_LIST}}` 和 `{{PERSONA_WEIGHTS}}` — 画像定义和权重
4. `{{ANALYSIS_DIMENSIONS}}` — 交叉分析维度（可增减）
5. `{{FILTER_DIMENSIONS}}` — 筛选器选项（与走查数据结构对齐）

### 扩展指南
- **新增板块**: 在 Prompt 的「报告要求」中增加 Block 5 描述即可
- **调整严重度权重**: 修改 Block 1 的权重公式段落
- **调整色彩风格**: 修改「全局 HTML 技术规范」中的配色系统
- **多语言支持**: 在 `<html lang="">` 和字体栈中调整
