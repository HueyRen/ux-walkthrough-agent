## 截图协议（CRITICAL — 必须严格遵守）

本次走查需要产出两类截图，缺一不可。

### 类型一：步骤截图（步骤截图）— 每步必须执行

**触发条件**：每完成一个操作（导航、点击、滚动、输入等）后，立即截取全页截图。

**命名规则**：
```
{round}_{station}_{step_number}_{action_description}.png
```
- `round`：走查轮次编号（如 `r1`、`r2`）
- `station`：当前子环节名称（如 `homepage`、`search_results`、`product_detail`）
- `step_number`：该 station 内的步骤序号（如 `s01`、`s02`）
- `action_description`：操作描述，英文小写 + 下划线（如 `navigate_to_homepage`、`click_search_button`、`scroll_down_500px`）

**格式要求**：
- 全页截图，不裁剪
- 无标注（原始页面状态）
- 作为走查全程的证据链

**示例**：
```
r1_homepage_s01_navigate_to_homepage.png
r1_homepage_s02_scroll_down_500px.png
r1_search_results_s01_enter_keyword.png
```

---

### 类型二：问题截图（问题截图）— 发现问题时执行

**触发条件**：识别到 UX 问题时，在当前可见视口截图并标注。

**命名规则**：
```
{round}_{station}_issue_{sequence}_{brief_description}.png
```
- `sequence`：问题序号（如 `01`、`02`）
- `brief_description`：问题简述，英文小写 + 下划线（如 `missing_cta`、`confusing_label`）

**标注要求**：
- 在问题元素上绘制**红色矩形框**，边框宽度 3px
- 标注区域面积：不超过视口的 50%，不小于 20×20px
- 如果问题元素需要滚动才能看见，先滚动到位，再截图标注
- 截图基于当前可见视口（不做全页截图）

**示例**：
```
r1_homepage_issue_01_no_search_placeholder.png
r1_search_results_issue_02_filter_hidden_on_mobile.png
```

## 新手用户认知约束（当画像为新手 / 新用户时，必须遵守）

当走查画像为**新手用户**或**平台新用户**时，以下行为限制强制生效：

1. **禁止提供搜索关键词** — 不得直接告知用户该输入什么词汇
2. **禁止直接 URL 导航** — 必须模拟用户从首页或入口自然点击进入
3. **平台专有术语 = 认知障碍** — 遇到平台特定词汇（如专属产品名、行业黑话、缩写），必须标注为认知障碍并在问题中记录
4. **模拟「什么都不知道」状态** — 不得利用对平台的先验知识跳过用户实际会遇到的困惑点

> 认知障碍标注示例：「用户在首页看到『一件代发』按钮时，无法理解该功能与普通采购的区别 — **[认知障碍]** 平台术语，新手无法理解」

## 通用工作规则

1. **真实访问，不凭想象** — 每条问题必须来自你实际访问到的页面状态
2. **保持画像视角** — 用该 persona 的认知水平、动机和行为模式去感知界面
3. **用用户语言描述问题** — 问题描述使用用户视角（「我不知道该点哪里」），不用技术语言（「该按钮缺少 onClick handler」）
4. **每条问题记录 URL** — 发现问题时立即记录当前页面完整 URL
5. **每个 Station 至少一张步骤截图** — 哪怕该 station 没有发现问题，也必须截取进入时的页面状态截图
6. **截图先于记录** — 先截图，再在问题记录中引用截图文件名
7. **按旅程顺序走查** — 不得乱序跳跃，确保覆盖 `{{JOURNEY_STAGES}}` 的每个阶段
